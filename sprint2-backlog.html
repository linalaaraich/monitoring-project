MVP Definition — what "done" looks like on April 9th

The AI RCA pipeline is the primary deliverable. The AWS demo environment is a temporary hosting vehicle for the April 9 demo only — all further development and production will be on CIRES private cloud.

  • Primary (Must): Grafana Alerting fires → triage service receives webhook → Drain3 annotates log anomalies → self-hosted Ollama autonomously gathers context via MCP bridges (Prometheus, Loki, Jaeger, Drain3, RCA History) → ESCALATE sends email with root cause analysis. Triage service is the sole notification gateway — no alert reaches a developer without AI evaluation. All layers present: metrics, logs, anomaly detection, LLM reasoning, notification. 5-min timeout fallback never drops an alert.
  • Supporting (Must): Full observability stack running on AWS EC2 for the demo (reusing existing docker-compose & Ansible). Small-scale sample app monitored. Reachable, scraping, dashboards green.
  • Post-demo: AWS resources terminated. Same Ansible playbooks reused on CIRES private cloud — only inventory IPs change.
Infrastructure note: AWS is for the April 9 demo only — resources will be terminated immediately after. The demo uses a production-like architecture: React frontend on S3 + CloudFront, MySQL on RDS, Spring Boot on a dedicated EC2, with Terraform provisioning all infrastructure. The monitored system is a small-scale sample app, not production workloads. Production deployment and all further development will be on CIRES private cloud infrastructure.

Epic 2 — AI RCA System (Sprint 2 MVP)

Three-layer AI pipeline: (1) Grafana Alerting + Drain3 detect anomalies → (2) FastAPI triage service receives webhooks, deduplicates, correlates, decides whether to investigate → (3) Ollama autonomously gathers context via MCP bridges (Prometheus, Loki, Jaeger, Drain3, RCA History) and reasons over enriched context → ESCALATE/DISMISS decision. Triage service is the sole notification gateway — sends email on ESCALATE via SMTP. All decisions stored in SQLite (RCA History). 5-min timeout fallback. All self-hosted, no external APIs.

E2-17
As the LLM, I can correlate data across metrics, logs, and traces within the same time window so that my root cause analysis connects quantitative signals (metric spikes), qualitative evidence (error logs), and request-level context (trace spans) into a coherent diagnosis.
Cross-pillar correlation: metrics ↔ logs ↔ traces via time window + trace_id
Implement correlation capabilities across MCP bridges that the LLM can leverage during investigation: (1) Time-based correlation: Given an alert at time T, the LLM queries Prometheus for metrics from T-15m to T, Loki for logs in the same window, and Jaeger for traces in the same window — all three data sources aligned to the same incident timeframe. (2) Trace-log correlation: Spring Boot logs include trace_id (injected by OTel Java Agent). The LLM can extract a trace_id from a Loki log line, then call Jaeger MCP's get_trace(trace_id) to see the full request path, identifying which span failed and where latency spiked. (3) Trace-metric correlation: The LLM can identify a failing service from trace error spans, then query Prometheus for that service's CPU/memory/error rate metrics in the same time window. (4) Log-metric correlation: When Drain3 flags anomalous log patterns, the LLM checks Prometheus to see if corresponding metric anomalies exist (e.g., error rate spike coinciding with new error log templates). The LLM prompt template instructs it to perform cross-pillar correlation as a standard investigation step: “Always check all three pillars (metrics, logs, traces) within the alert time window before reaching a verdict.”

Risk Register

High Risks

  • AWS account provisioning delayed by supervisor/admin. Mitigation: submit cost justification doc immediately. Start Terraform module development and triage service development locally while waiting.
  • Ollama cold-start or OOM on g4dn.xlarge. Mitigation: pre-pull model during setup. Use quantized (Q4) variant of llama3.1:8b. Test memory footprint before integrating. T4 GPU has 16 GB VRAM — 8B model (Q4) fits in ~5 GB.
  • LLM produces inconsistent JSON output. Mitigation: strict output format instruction + Pydantic validator + one retry + ESCALATE default. Build and test the parser before integrating Ollama.

Medium Risks

  • AWS costs exceed expectations. Billing alerts at $100/$200/$300. Stop all instances outside work hours. Small-scale demo keeps costs low (~$82–$240 total). Terraform makes teardown trivial (terraform destroy). Terminate all resources immediately after demo.
  • LLM RCA quality is poor with 8B model. Try Mistral 7B as alternative. Iterate on prompts. For demo, pre-select alert scenarios that produce good results.
  • Network latency between VMs affects triage pipeline. All VMs in same VPC/AZ. Context gathering timeout (8s per source) handles this.
  • NVIDIA driver installation on g4dn.xlarge. Use Amazon Deep Learning AMI which has drivers pre-installed. Fallback: manual CUDA toolkit install.
Future sprints (on CIRES private cloud): Private cloud deployment, WATCH state re-checker (persistent queue with APScheduler), deep prompt engineering, advanced MCP tooling (system metadata, deployment history), and full AI pipeline test suite. AWS is not part of any future sprint — all further work is on private cloud. Kubernetes is permanently out of scope.
