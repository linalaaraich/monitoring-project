Key Data Flows

METRICS Prometheus ← scrape → all exporters (15s)
TRACES OTel Java Agent → OTel Collector :4317 (gRPC) → Jaeger
TRACES Kong OTel Plugin → OTel Collector :4318 (HTTP) → Jaeger
LOGS OTel Collector filelog (Backend/Net VMs) → Loki :3100 (with trace_id extraction)
VIZ Grafana → Prometheus / Loki / Jaeger
TRAFFIC End User → Kong :8000 → Spring Boot :80 (/api, /actuator) | / → CloudFront (S3)
ALERT Grafana Alerting → Triage Service :8090 (webhook)
AI Triage Service → Ollama :11434 (LLM inference)
MCP Ollama → MCP Bridges (:8091-8095) → Prometheus / Loki / Jaeger / Drain3 / History
EMAIL Triage Service → SMTP (Gmail) → Developer inbox
IDTypeFromToProtocolDescription
df-1 Metrics Prometheus node_exporter (all VMs), cAdvisor (all VMs), Spring Actuator (via Kong :8000), Kong Admin :8001, Grafana, Loki, Jaeger, OTel Collector HTTP GET Prometheus scrapes metrics from all targets every 15s
df-2 Traces OTel Java Agent (Spring Boot) OTel Collector :4317 gRPC (OTLP) Spring Boot emits traces via auto-instrumented OTel Java agent
df-3 Traces Kong OpenTelemetry Plugin (per-service) OTel Collector :4318 HTTP (OTLP) Kong gateway emits W3C traces via per-service OpenTelemetry plugin (extract: w3c + b3)
df-4 Traces OTel Collector Jaeger :4317 (internal Docker network) gRPC (OTLP) Collector exports traces to Jaeger via internal Docker network (host port :4327)
df-5 Logs OTel Collector filelog (Backend VM) Loki :3100 HTTP POST (Loki exporter) OTel Collector tails /var/log/app/*.log, extracts trace_id, ships to Loki
df-6 Logs OTel Collector filelog (Network VM) Loki :3100 HTTP POST (Loki exporter) OTel Collector tails /var/log/kong/*.log, ships Kong logs to Loki
df-7 Visualization Grafana Prometheus :9090, Loki :3100, Jaeger :16686 PromQL / LogQL / HTTP Grafana queries all three observability backends
df-8 Traffic End User Kong :8000 HTTP End users access the application through Kong proxy
df-9 Traffic Kong :8000 Spring Boot :80 (Backend EC2) | CloudFront (S3) HTTP Kong routes /api, /actuator → Backend EC2; / → CloudFront (S3 React frontend)
df-10 Alert Grafana Alerting (Monitoring VM) Triage Service :8090 (AI/LLM VM) HTTP POST (webhook) Grafana evaluates alert rules and sends webhook to triage service — sole alert gateway
df-11 Alert Drain3 (AI/LLM VM) Triage Service :8090 (localhost) HTTP POST (webhook) Drain3 detects novel log patterns or anomaly rate spikes, fires webhook to triage service
df-12 AI Triage Service (AI/LLM VM) Ollama :11434 (AI/LLM VM) HTTP (Ollama API) Triage service calls Ollama for LLM-powered root cause analysis when alert passes triage
df-13 MCP Ollama (AI/LLM VM) MCP Bridges :8091-8095 (AI/LLM VM) MCP tool calls LLM autonomously queries Prometheus, Loki, Jaeger, Drain3, and RCA History via MCP bridges
df-14 Email Triage Service (AI/LLM VM) Gmail SMTP → Developer inbox SMTP (TLS) Sends validated RCA report to developers only when LLM confirms alert is valid
